# 引き継ぎドキュメント作成

このセッションのコンテキストを**一切持たない**次のセッションが、会話ログを読み返さずに作業を再開できる引き継ぎドキュメントを作成する。読み手は「このセッションで何が起きたか知らない有能な同僚」と想定すること。

## 手順

1. **事実確認を先に行う**（記憶だけで書かない）:
   - gitリポジトリなら `git status` と `git diff --stat` で変更ファイルを確認し、`git log --oneline -10` で直近のコミットを確認する
   - このセッションで実行したテスト・ビルドの最終結果（成功/失敗）を確認する
2. 下のテンプレートに**そのままの見出し構成で**従い、HANDOVER.md を書く。該当する内容がない節は削除せず「なし」と書く
3. **保存場所**: gitリポジトリならそのルート、そうでなければ現在の作業ディレクトリに `HANDOVER.md` として保存する
4. 既に HANDOVER.md が存在する場合は上書きする。ただし旧内容にまだ有効な未完了タスクがあれば、新しい「未完了タスク」節に統合してから上書きする

## テンプレート

```markdown
# 引き継ぎ: <作業内容を一言で>

- **日付**: <YYYY-MM-DD>
- **ブランチ / コミット**: <ブランチ名> @ <短縮ハッシュ>（gitでなければ「なし」）

## 目的
<ユーザーの元々の依頼。途中で方針変更があればそれも1〜2文で>

## 行った作業
<変更ごとに: 何を・どのファイルで・なぜ。例:
- `src/auth.ts:120` の `refreshToken()` — トークン更新の競合状態を修正（同時リクエストで二重更新されていた）>

## 決定事項
<決定 + 理由 + 検討して却下した代替案。例:
- キャッシュはRedisではなくインメモリを採用 — デプロイ構成を増やしたくないため。Redis案はスケール要件が出たら再検討>

## 未完了タスク
<チェックボックス形式。各タスクに「次にやる具体的な一手」（実行するコマンド、編集するファイル）を必ず書く。例:
- [ ] E2Eテストの修正 — 次の一手: `npm run test:e2e` を実行し、`tests/login.spec.ts` の失敗を確認する>

## 落とし穴と教訓
<ハマったこと + 原因 + 回避方法。例:
- `npm test` がwatchモードで固まる → CI用には `npm test -- --run` を使う>

## 環境・再開方法
<ビルド・テスト・起動の主要コマンド。特殊なセットアップ（環境変数、要ログインなど）があれば明記>
```

## 品質基準（書き終えたら全項目をチェックする）

- **固有名詞で書く**: 「バグを修正した」ではなく「`src/auth.ts` の `refreshToken()` の競合状態を修正した」。ファイルは `パス:行番号` 形式で参照する
- **このセッション内でしか通じない略称・コードネーム・「例のやつ」を使わない**。初出の用語はその場で説明する
- **相対日付を使わない**: 「昨日」「来週」ではなく絶対日付（YYYY-MM-DD）で書く
- **未完了タスクには必ず「次の一手」がある**: 読んだ人がコマンド1つ・ファイル1つから着手できること
- **失敗も正直に書く**: テストが落ちたまま・未検証の変更があるなら、その旨を明記する
- 最終チェック: **「会話ログを一切読まずに、この文書だけで作業を再開できるか？」** — できないなら足りない情報を追記する

作成後、保存先のフルパスと、未完了タスクの件数をユーザーに報告する。
