# 体系的デバッグ

推測で修正せず、証拠で原因を特定してから最小の修正を行う。以下のフェーズを**順番に**実行し、フェーズを飛ばさない。

## フェーズ1: 再現

1. 「期待する動作」と「実際の動作」をそれぞれ1文で明文化する
2. 問題を再現する最小のコマンドまたは手順を確立する(再現できないバグは直せない)
3. 再現できない場合: 完全なエラーメッセージ・発生条件(環境、タイミング、入力)をユーザーに確認する。**推測での修正はしない**

## フェーズ2: 原因特定

1. エラーメッセージとスタックトレースを**最後まで**読む(先頭行だけで判断しない)
2. 仮説を立てる前に事実を集める: 該当コードを読む、`git log --oneline -10 -- <file>` で最近の変更を確認する、ログを見る
3. 仮説は1つずつ検証する。検証方法は先に「この仮説が正しければ X が観測されるはず」の形で決めてから実行する
4. 仮説が3回外れたら、前提を疑う: そもそもそのコードは呼ばれているか？ 実行されているのは最新のビルドか？ キャッシュや環境差ではないか？

## フェーズ3: 修正

- 原因に対する**最小の修正**を行う。ついでのリファクタリングや無関係な改善はしない
- 「症状を消す」修正(例外を握りつぶす、条件分岐で回避)になっていないか自問する。原因を直せない事情があるなら、その旨を明記して回避策として提示する

## フェーズ4: 検証

1. フェーズ1の再現手順を再実行し、問題が解消したことを確認する
2. 既存のテストを実行し、回帰がないことを確認する
3. 可能なら再発防止のテストを追加する(このバグを再現する入力でテストを書く)

## 報告(最後に必ずこの3項目で報告する)

- **原因**: なぜ起きたかまで含めて1〜2文。例:「`src/cache.ts:45` でTTL比較がミリ秒と秒を混同しており、キャッシュが即時失効していた」
- **修正**: どのファイルをどう変えたか
- **検証**: 実行したコマンドと結果。テスト未実行・未再現のまま終える場合は「未検証」と明記する

## 禁止事項

- 再現確認せずに「おそらくこれで直ります」と修正だけ出すこと
- 原因不明のまま複数箇所を同時に変更すること
- エラーを握りつぶして症状だけ消すこと
- 検証していないのに「修正しました」と断定して報告すること
