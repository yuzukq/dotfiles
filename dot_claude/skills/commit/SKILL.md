# コミット作成

変更内容を確認してから、リポジトリの規約に沿ったメッセージでコミットする。**確認より先にコミットしない。**

## 手順

1. **状態確認**(必ず最初に、まとめて実行する):
   - `git status` — 変更・未追跡ファイルの一覧
   - `git diff` と `git diff --cached` — 実際の変更内容
   - `git log --oneline -10` — このリポジトリのコミットメッセージのスタイル(言語、prefix規約の有無)
2. **ブランチ確認**: main / master などデフォルトブランチ上にいる場合は、先に作業ブランチを作成する(例: `git switch -c fix/cache-ttl`)
3. **含めるファイルの選別**:
   - 今回の作業に関係するファイルだけを `git add <path>` で個別に追加する。**`git add -A` や `git add .` は使わない**
   - 除外するもの: 一時ファイル、デバッグ用の print/console.log、`.env` などの秘密情報、無関係な自動生成差分。diff に秘密情報(APIキー、パスワード、トークン)が含まれていたらコミットを中断してユーザーに報告する
4. **粒度の判断**: 変更に無関係な作業が2つ以上混ざっている場合(例: バグ修正+新機能)は、別々のコミットに分割する
5. **メッセージ作成**: 手順1で確認した既存スタイルに合わせる。既存スタイルが読み取れない場合のデフォルト:
   - 1行目: 変更の要約を50字以内で「何をしたか」(例: `fix: キャッシュTTLの単位混同を修正`)
   - 本文(変更が自明でない場合のみ): 「なぜ」を1〜3行で
6. **コミット実行**: pre-commit フックが失敗した場合は、原因を修正して1回だけ再試行する。再度失敗したらエラー内容をユーザーに報告する。**`--no-verify` でフックを回避しない**
7. push はユーザーに明示的に頼まれた場合のみ行う

## 報告(最後に必ず)

- コミットハッシュとメッセージ
- 含めたファイル数と、意図的に**含めなかった**変更があればその一覧と理由

## 禁止事項

- diff を確認せずにコミットすること
- `git add -A` で無差別にステージすること
- `--no-verify` / `--force` を無断で使うこと
- 頼まれていない push・amend・rebase をすること
