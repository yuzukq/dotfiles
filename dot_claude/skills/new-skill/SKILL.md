# Skill作成(Sonnet最適化パターン)

モデルの推論力に頼らず、指示の明示性で品質を担保するSkillを作る。原則: **大きいモデルが暗黙にやることを、全部文章にする。**

## 手順

1. **要件確認**: そのSkillが「いつ発動し」「何を入力に」「何を出力するか」を1文ずつで確定する。ユーザーの依頼から読み取れない場合のみ質問する
2. **保存場所の決定**:
   - どのプロジェクトでも使う汎用作業 → `~/.claude/skills/<name>/SKILL.md`(個人用)
   - 特定リポジトリ固有の作業 → そのリポジトリの `.claude/skills/<name>/SKILL.md`(プロジェクト用)
   - `<name>` は小文字とハイフンのみ。frontmatter の `name` とディレクトリ名を一致させる
3. **下のスケルトンに沿って書く**。5つの必須要素をすべて入れる:
   - **固定テンプレート**: 出力がドキュメントなら見出し構成をコードブロックで丸ごと与える(構成をモデルに考えさせない)
   - **番号付き手順**:「事実確認 → 実行 → 検証 → 報告」の順。確認より先に行動させない
   - **実例の埋め込み**: 期待する粒度を抽象的な指示ではなく具体例で示す(「固有名詞で書く。例: `src/auth.ts:120` の `refreshToken()` の競合状態を修正」)
   - **エッジケースの事前決定**: 既存ファイルの扱い、対象が見つからない場合、失敗時の再試行回数など、実行時に迷いそうな判断を先に決めておく
   - **禁止事項**: 小さいモデルがやりがちな手抜き(未検証で完了報告、確認せず推測で断定、無差別な一括操作)を明示的に禁止する
4. **description を書く**: 発動判定に使われるので、機能の説明に加えて**ユーザーが実際に言いそうなトリガーフレーズ**を含める(例:「『テストを書いて』『テスト追加して』と言われたときに使う」)
5. **検証**: 作成したSKILL.mdを読み返し、下の品質チェックリストを全項目確認する。ユーザーには保存先パスと使い方(`/<name>`)を報告する

## スケルトン

````markdown
---
name: <skill-name>
description: <何をするか1文> + <「〜して」と言われたときに使う、というトリガーフレーズ>
---

# <タイトル>

<目的と大原則を1〜2文。例: 「推測で修正せず、証拠で原因を特定してから最小の修正を行う」>

## 手順

1. **事実確認**(必ず最初に): <実行するコマンド、読むファイルを具体的に>
2. <実行ステップ。判断基準は「〜の場合は〜する」の形で書く>
3. **検証**: <完了をどう確認するか>

## テンプレート(出力がドキュメントの場合のみ)

```markdown
<見出し構成を固定し、各節にプレースホルダと実例を書く>
```

## 報告(最後に必ず)

- <報告する項目を固定する。例: 変更ファイル、実行コマンドと結果、やり残し>

## 禁止事項

- <このタスクでモデルがやりがちな手抜き・危険行動を3〜5個>
````

## 品質チェックリスト(作成後に全項目確認する)

- [ ] description にトリガーフレーズが入っているか
- [ ] 手順が「事実確認 → 実行 → 検証 → 報告」の順になっているか
- [ ] 各指示に具体例があるか(抽象的な「適切に」「ちゃんと」が残っていないか)
- [ ] 実行時に迷う判断(既存ファイル、失敗時、対象なし)が事前に決められているか
- [ ] 禁止事項に「未検証で完了報告しない」が含まれているか
- [ ] 全体が150行以内か(長すぎる指示は守られない。溢れる分は参照ファイルに分離する)

## 禁止事項

- ビルトインスキル(code-review, simplify, verify 等)と同じ機能のSkillを作ること(先に既存スキル一覧を確認する)
- テンプレートなしで「いい感じにまとめて」型のSkillを作ること(それはSonnetで品質が落ちる典型)
- 検証手順のないSkillを作ること
