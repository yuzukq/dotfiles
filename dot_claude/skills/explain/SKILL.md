# コード解説

コードを実際に読んで確認した事実だけで解説を書く。**推測で仕様を語らない。**

## 手順

1. **読む範囲を決める**: 対象のエントリポイント(公開関数、ルート定義、main)を特定し、そこから呼ばれる主要な処理を追う。関連する設定ファイル・型定義も確認する
2. **裏取り**: 解説に書く各主張について、根拠となるコード箇所(`パス:行番号`)を特定する。コードで確認できないこと(歴史的経緯、設計意図)は「コードからは読み取れない」と明記するか、git log で確認する
3. **下のテンプレートで解説を書く**。出力は会話への返信として書き、ファイル保存はユーザーに頼まれた場合のみ行う

## テンプレート

```markdown
## 一言でいうと
<このコード/モジュールの役割を1〜2文で。例:「HTTPリクエストのレート制限をIPアドレス単位で行うミドルウェア」>

## 全体像
<主要な構成要素とその関係を3〜6項目で。各項目に `パス:行番号` を添える>

## 主要な処理の流れ
<代表的な1リクエスト/1呼び出しが通る経路を、番号付きステップで。
例:
1. `middleware.ts:12` でリクエストを受け、IPを抽出
2. `store.ts:30` のカウンタをインクリメント
3. 上限超過なら `middleware.ts:25` で429を返す>

## 注意点・落とし穴
<読み手がハマりそうな点: 暗黙の前提、紛らわしい命名、非自明な副作用、既知の制限。なければ「特になし」>

## 関連ファイル
<今回触れたファイルの一覧を1行説明付きで>
```

## 品質基準(書き終えたらチェックする)

- すべての主張に `パス:行番号` の根拠があるか。**行番号は実際に読んで確認したものだけを書く**(推測の行番号は書かない)
- 対象読者に合わせているか: 指定がなければ「このリポジトリを初めて見る有能なエンジニア」を想定し、リポジトリ固有の用語は初出時に説明する
- コードの動作(事実)と自分の解釈(推測)を区別して書いているか。推測には「〜と思われる」を付ける
- 長さは対象の複雑さに比例させる: 関数1つなら数段落、モジュール全体ならテンプレート全節。**水増ししない**

## 禁止事項

- コードを読まずに名前から動作を推測して断定すること
- 存在しない関数・ファイルへの言及(必ず実在を確認する)
- 網羅性のためだけの長大なファイル一覧(解説に必要なものだけ)
