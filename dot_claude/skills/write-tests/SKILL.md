# テスト作成

既存のテスト規約に合わせ、テストマトリクスを先に作ってから書く。**いきなりテストコードを書き始めない。**

## 手順

1. **既存規約の確認**(必ず最初に):
   - テストフレームワークとテストの実行コマンドを特定する(package.json / pyproject.toml / Makefile 等)
   - 対象コードに最も近い既存テストファイルを1つ以上読み、命名・配置・セットアップ・アサーションのスタイルを把握する
   - 既存テストが1つもない場合: フレームワークの標準構成で最小のセットアップを作り、その旨をユーザーに報告する
2. **対象の理解**: テスト対象のコードを読み、入力・出力・副作用・エラー条件を列挙する
3. **テストマトリクスを作る**: コードを書く前に、以下の3分類でケースを列挙する:

   | 分類 | 観点 | 例(`parseAmount(s: string): number` の場合) |
   |---|---|---|
   | 正常系 | 代表的な入力 | `"1000"` → `1000` |
   | 境界 | 空・0・最大・重複・順序 | `""`、`"0"`、`"999999999999"`、先頭ゼロ |
   | 異常系 | 不正入力・失敗パス | `"abc"` → 例外、`null` → 例外、負数の扱い |

   副作用がある関数は「呼ばれること/呼ばれないこと」の検証も含める
4. **実装**: マトリクスの各行を1テストケースにする。テスト名は「何をしたら何になるか」が読めるようにする(例: `空文字列を渡すとValidationErrorを投げる`)
5. **実行と確認**:
   - テストを実行し、全件通ることを確認する
   - 新規テストが「何もテストしていないテスト」でないか確認する: 対象の振る舞いが壊れたら落ちるか？ 疑わしい場合は一時的に実装を1箇所壊して落ちることを確認し、**必ず元に戻す**
   - 落ちたテストを、実装ではなく期待値の方を書き換えて通すのは禁止。実装のバグを見つけた場合はテストをそのままにして、バグとしてユーザーに報告する

## 報告(最後に必ず)

- 追加したテストファイルとケース数(正常系/境界/異常系の内訳)
- テスト実行コマンドと結果(全件パス or 失敗の詳細)
- テストで発見した実装のバグがあればその内容

## 禁止事項

- 既存テストのスタイルを確認せずに独自流で書くこと
- 実行せずに「テストを追加しました」と報告すること
- 通すために期待値をコピペで合わせること(スナップショットの無思考な更新を含む)
- モックだらけで実質何も検証していないテストを書くこと
